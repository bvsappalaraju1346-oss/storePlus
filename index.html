01
Your biggest risk isn't a competitor — it's Electronics.
Electronics brings in $178K (58% of total). That's great until a supply disruption, a price war, or a viral review tanks it. You need Clothing at 30%+ and Sports needs an emergency promotion. Right now Sports = $1,138. That's not a category — that's a rounding error.
→ Fix category concentration
02
111 customers bought once and disappeared.
A post-purchase email sequence with a 10% second-order coupon could realistically recover 25–30 of them. At $137 AOV that's $3,400–4,100 in recaptured revenue — essentially free money sitting in your database.
→ Launch win-back campaign
03
Three customers = nearly a third of your revenue.
CUST0155, CUST0065, CUST0203 together spent ~$98K. If one of them leaves, that's a 32% revenue cliff. They need VIP treatment — not a newsletter.
→ Build VIP tier now
04
Gym Bag sold 1 unit. One.
Gym Bag ($57), Plant Pot ($87), Comic Book ($251) are all near-dead. Bundle them with top sellers or remove them. Dead inventory has a real carrying cost most stores ignore.
→ Audit dead stock
05
You're leaving $12K on the table at checkout.
AOV is $137 but your target is $150. A "Frequently Bought Together" widget hitting 30% of orders at a $15 upsell = ~$12,000/year. Not a big engineering project — a one-week feature.
→ Implement cross-sell
06
Stock up in July. August will thank you.
August is your peak month at $34.1K — 56% above February's low of $19.8K. If you're not pre-stocking in mid-July and running early-bird promotions by July 20, you're missing demand you already know is coming.
→ Plan inventory calendar
07
Free shipping at $150 is a clever trick.
30% of orders paid $12.99 shipping. Set a free shipping threshold at $150 (just $13 above AOV). Customers instinctively add items to hit thresholds — it's one of the highest-ROI levers in e-commerce.
→ Set free shipping threshold

Known limitations.
Honest about what this project can't do — and what comes next
📝
A note from the developer
I'm fully aware of these limitations — this is version 1.0 of StorePulse, built as a portfolio project to demonstrate end-to-end data analysis and dashboard development. The data is synthetic, the dashboard is desktop-only, and there's no live backend yet. I plan to fix all of these in the next version — starting with connecting real data, adding mobile support, and building a forecasting layer. This page exists because I believe in being transparent about what a project can and can't do.
🛠
Why show limitations?
Most dashboards pretend to be perfect. This one doesn't. Knowing what a tool can't do is just as important as knowing what it can. These are the real gaps — and honest next steps to fix each one.
🗄️
Data is hardcoded — not live
High Priority
All 2,500 orders are embedded directly inside this HTML file. In a real product, the dashboard would connect to a live database (PostgreSQL, Firebase) or pull from an API every hour. Right now, if the data changes — someone has to manually edit the code.
NEXT STEP →
Connect to a Google Sheets API or a REST backend. That way the dashboard refreshes automatically whenever new orders come in.
🎲
Dataset is synthetic, not real
Medium Priority
The 2,500 orders were generated using Python with realistic patterns — seasonal peaks, skewed customers, random discounts. But real data is messier: duplicate customers with different spellings, missing zip codes, currency mismatches, refund edge cases. The cleaning pipeline here is simpler than what real data demands.
NEXT STEP →
Upload a real CSV from Shopify, WooCommerce, or Amazon Seller Central and re-run the full Python cleaning + analysis pipeline.
📱
Not mobile friendly
Medium Priority
The sidebar, chart grids, and table layouts are built for desktop screens (1200px+). On a phone or tablet, the sidebar collapses awkwardly, charts become too small to read, and the filter bar wraps badly. A business owner checking numbers on their phone would have a frustrating experience.
NEXT STEP →
Add a hamburger menu for mobile, stack cards to single column below 768px, and increase chart touch targets for finger navigation.
🔐
No login or authentication
Medium Priority
Anyone with the GitHub Pages URL can view the full dashboard — all revenue figures, customer IDs, and business data. For a demo project this is fine, but for a real business with sensitive financial data, you'd need authentication (login page, session tokens, role-based access) before deploying publicly.
NEXT STEP →
Add password protection via Netlify Identity or move to a backend framework (Flask, Node.js) with session-based auth.
🔮
No predictive analytics
Future Feature
This dashboard shows what happened — it doesn't predict what will happen. There's no revenue forecasting, no churn prediction, no "if we run a 15% discount next month, here's the expected impact." That kind of forward-looking analysis requires ML models (linear regression, Prophet, XGBoost) on top of the existing data pipeline.
NEXT STEP →
Use Facebook Prophet or scikit-learn to add a "Forecast" tab showing next 3 months projected revenue with confidence intervals.
📤
Export limited to CSV only
Future Feature
Right now export downloads a CSV file — useful, but limited. A real analytics tool would also export a formatted PDF report with charts included, push data to Google Sheets automatically, or email a weekly summary to stakeholders. The share button currently does nothing.
NEXT STEP →
Use jsPDF + html2canvas to export the dashboard as a PDF. Add a scheduled email report via a simple Node.js cron job.
